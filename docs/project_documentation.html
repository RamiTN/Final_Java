1. Project Architecture Overview

How all layers are organized and communicate with each other

This project follows a classic MVC (Model-View-Controller) pattern enriched with a DAO (Data Access Object) layer and a Service layer. This keeps each piece of code focused on one job only.

Layer Responsibilities

LayerPackageWhat it does
ModelmodelPlain Java objects (POJO) that hold data. No logic, just fields + getters/setters.
DatabasedbCreates the SQLite connection and initializes tables on first launch.
DAOdaoAll SQL queries live here. Each DAO handles one table. Returns Model objects.
ServiceserviceBusiness logic: authentication, PDF export, AI calls. Talks to DAOs, not DB directly.
ControllercontrollerHandles UI events (button clicks). Calls Services/DAOs and updates the View.
ViewviewFXML XML files that define the visual layout. Linked to Controllers.
💡 Key Design Rule: Controllers never touch the database directly. They call a Service or a DAO. Services hold reusable business logic. DAOs hold all SQL. This makes the code clean, testable, and easy to extend.

2. Main.java – Application Entry Point

The class that boots the entire application

Entry Point
class

Main

Package: default (root)

Extends: javafx.application.Application

This is the very first class that runs when you launch the program. It extends JavaFX's Application class, which is the standard way to start a JavaFX desktop app. It does three important things in sequence:

  1. Calls DatabaseConnection.initializeDatabase() to create the SQLite database and tables if they don't exist yet.
  2. Loads landing.fxml as the first screen using FXMLLoader.
  3. Creates the main window (Stage), sets the title to "CV Creator", maximizes it, and shows it.
MethodReturnDescription
start(Stage primaryStage)voidJavaFX lifecycle method. Initializes DB, loads landing.fxml, sets up and shows the main window at 800×600 (maximized).
main(String[] args)voidStandard Java entry point. Calls launch(args) which triggers JavaFX to call start().
💡 Why extend Application? JavaFX requires a class to extend Application and override start(). The launch() method internally creates the JavaFX thread, loads the scene graph, and calls start() on the JavaFX Application Thread.

4. Database Layer – DatabaseConnection.java

The single point of contact between the application and the SQLite file

db
class

DatabaseConnection

Package: db  |  File: cvcreator.db (SQLite)

This utility class has two jobs: provide a database connection to whoever needs it, and create all tables on first launch. It uses the JDBC SQLite driver from Xerial.

MethodReturnDescription
getConnection()ConnectionStatic. Opens and returns a new JDBC connection to cvcreator.db. Every DAO calls this method each time it needs to run a SQL query. Uses try-with-resources to auto-close connections.
initializeDatabase()voidStatic. Called once at startup by Main.java. Creates three tables (users, cvs, jobs) using IF NOT EXISTS so it's safe to call every time. Also inserts the default admin account if none exists.
🔑 Default Admin Account: On first launch, if no user with email admin@admin.com exists, the system automatically inserts one with password admin123 and role admin. This ensures there's always a way to access the admin dashboard.

Connection URL

The connection string is: jdbc:sqlite:cvcreator.db — this means SQLite will look for a file named cvcreator.db in the working directory (the project root).

6. Service Layer

Business logic that sits between Controllers and DAOs

Services contain logic that is too complex for a DAO but too reusable to put inside a specific controller. They are the "brain" of the application.

service

AuthService.java

Purpose: Manages the currently logged-in user as a global session. Handles login and registration logic.

Key design: currentUser is a static field — it exists once for the entire application lifetime. This acts as a simple in-memory session, since this is a desktop app (no HTTP sessions needed).

How exportToPdf() Works

  1. Creates a new PDDocument with one A4 page.
  2. Draws a dark-blue header bar across the top of the page.
  3. Writes the full name (centered, large white text) and contact info (email | phone | address).
  4. Calls addSection() for each section: Objective, Education, Experience, Skills, Languages.
  5. Each section has a bold colored title, a thin underline, and word-wrapped body text.
  6. Saves the document to the chosen file path.

7. Controller Layer

Connecting the FXML screens to the Services and DAOs

Every screen (FXML file) has one Controller class. Controllers respond to button clicks, read what the user typed in the forms, and update the labels and tables on the screen.

controller

LandingController.java

Screen: landing.fxml (The very first screen)

What it does: Displays a beautiful welcome screen. It uses all 3 DAOs to count total users, CVs, and Jobs to display dynamic statistics. It also loads the 5 most recent job offers and displays them as clean cards on the screen.

controller

UserController.java

Screens: login.fxml, register.fxml, dashboard.fxml, profile.fxml

What it does:

  • Auth: Handles the logic when the user clicks Login or Register. Validates emails, passwords, and name formats. Calls AuthService.
  • Dashboard: Populates the user dashboard with their recent CVs, calculates a "Profile Completion %", and shows how many jobs exist.
  • Search: Includes a real-time search listener that filters the user's recent CVs by name, skills, or objective as they type.
  • Navigation: Has many goToX() methods to load other screens.
controller

CVController.java

Screen: cv.fxml (The CV Editor)

This is the largest controller. It manages a dynamic form where users can create, edit, delete, and download CVs.

Key Features:

  • Dynamic Lists: Users can click "Add" to spawn new TextFields for Education and Experience. The controller dynamically creates HBox UI elements in Java.
  • Skill Tags: When adding a skill, it generates a little blue "tag" with an X button to remove it.
  • Form Validation: Checks for minimum name length, valid email regex, and valid phone number patterns before saving.
  • Data Conversion: Takes the visual dynamic lists and packs them into single Strings (separated by \n---\n) to give to the CvService for DB storage.
  • AI Integration: Has a button that sends the current form data to AiService and asks Gemini to suggest improvements for the CV.
  • PDF Export: Triggers a FileChooser dialog to let the user pick where to save the PDF, then calls CvService.exportToPdf().
controller

JobGuideController.java

Screen: JobGuide.fxml (The AI Career Assistant)

A chat interface powered entirely by Gemini AI. It has three distinct modes:

  1. Standard Chat: Acts as a general career coach.
  2. Interview Practice: An interactive loop. The AI asks a predefined question ("Tell me about yourself"). The user types an answer. The AI evaluates the answer, provides feedback and an example, and asks the user to try again. Only when the user types "next" does it move to question 2.
  3. CV Advice: Reads the user's first CV from the database and asks the AI to review it.
  4. Job Matcher (No AI): A local algorithm that takes the skills from the user's CV and checks if they exist in any Job descriptions/requirements in the database.

Uses multi-threading so the UI doesn't freeze while waiting for the Gemini API.

controller

AdminController.java

Screen: AdminDashboard.fxml

Only accessible by users with the "admin" role. It manages the whole system:

  • User Management: TableView of all users. Admins can delete users (but not themselves).
  • Job Management: Full CRUD (Create, Read, Update, Delete) forms for job postings. Includes URL validation for job links.
  • Charts: Uses JavaFX PieChart to show the ratio of Admins vs Regular Users, and a BarChart to show the number of jobs per company.
controller

JobController.java

Screen: Job.fxml

A very simple read-only controller for regular users. It loads all jobs from JobOfferDAO into a TableView so users can browse available positions.

controller

InfoController.java

Screens: features.fxml, how_it_works.fxml

A purely navigational controller used by the static informational pages attached to the landing screen. Contains only goToX() methods.

10. Application Flow & User Journey

  1. Launch: Main.java ensures DB exists. Shows Landing Page.
  2. Landing Page: Shows stats & recent jobs. User clicks "Login".
  3. Login: User logs in. AuthService saves session.
    • If Role == Admin ➔ Goes to Admin Dashboard (Manage users & jobs).
    • If Role == User ➔ Goes to User Dashboard.
  4. Dashboard: Shows recent CVs and profile completion. User clicks "Create CV".
  5. CV Editor: User fills out forms, adds dynamic rows for Education/Experience, adds tags for Skills.
    • Action: Clicks AI Advice ➔ Gemini analyzes fields and gives tips.
    • Action: Clicks Save ➔ Saves to SQLite DB.
    • Action: Clicks Export PDF ➔ PDFBox generates a formatted file.
  6. Job Guide: User clicks the AI icon.
    • Chats with Gemini for career advice.
    • Practices an interactive interview loop.
    • Gets matched with internal jobs based on their CV skills.
